# 贴纸成就系统合并设计

## 合并原则

- `calm-shopping-main-new` 是唯一主线；新版账号、头像、个人资料、首页快捷入口和持久化逻辑全部保留。
- 旧目录只作为贴纸二期功能来源，不整文件覆盖新版的 `index.html`、`main.js`、`store.js` 或 `auth.js`。
- 二期继续复用现有 12 张贴纸和 3 个场景，不迁移 v1 布局。

## 文件与接入点

- 原样移植贴纸素材、编辑器、贴纸墙、目录、模型、所有权、奖励队列、云端布局、测试和 Supabase migration。
- 在新版小院中用贴纸大厅替换静态图鉴，保留新版个人中心和其他页面结构。
- 在新版 `main.js` 的报告生成、决策确认、加入沉淀池、放弃购买、登录初始化处接入幂等成就事件。
- 在新版 `store.js` 增加河币队列冲刷、RPC 返回余额落地和无递归事件刷新，不改变已有 companion/profile 数据读取。
- 新版 `auth.js` 保持原样；贴纸模块只使用其现有 `sb()`、`currentUserId()` 和 `onChange()` 接口。

## Supabase 部署

- 从新版前端配置自动解析 project ref，使用用户本机环境变量中的 Personal Access Token 调用 Management API。
- 先以只读 SQL 核验 `user_stats.user_id/coins` 及现有策略，再执行 migration。
- migration 事务失败时不留下半成品；成功后查询五张表、四个 RPC 和 RLS 状态。
- 不读取、不输出、不写入仓库任何 token；部署结束后删除用户级 `SUPABASE_ACCESS_TOKEN`。

## 验证

- 先在新目录运行二期测试并确认缺失功能导致失败，再完成合并直到测试通过。
- 回归新版后端 17 项测试、全部前端脚本语法检查。
- 浏览器检查游客仅 1 张默认贴纸、贴纸墙 12 张卡片、编辑器仅显示拥有贴纸、430px 无溢出、桌面 393×852 手机壳。
- 数据库检查重复事件、重复购买、余额不足、跨账号 RLS 和奖励恢复；不使用生产用户真实购买数据做破坏性测试。
