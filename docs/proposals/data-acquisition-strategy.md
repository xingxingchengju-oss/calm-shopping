# 提案 · 口碑佐证的数据获取策略（已采纳：方案 B）

> **状态：已采纳方案 B（2026-06）。** 已落地为 `backend/app/services/insight` + `backend/app/integrations/{review_search,xhs_scraper}`，可运行：`cd backend && python app/services/insight/demo.py 破壁机`。
>
> 背景：PRD §5.4「真实口碑佐证」需要「该不该买」的真实信息。本文汇总实测证据 + 候选方案 + 最终决定。

## 1. 实测证据（2026-06，本环境真实请求）

| 目标 | 结果 |
|------|------|
| 原仓库 `web_api/sns/*` 端点 | HTTP 500，已失效 |
| 小红书 关键词搜索 | 结果走前端签名 XHR，**不在 SSR**，匿名为空 → 需登录 Cookie |
| 小红书 笔记**正文** | ✅ 公开可抓（需该笔记 `xsec_token`）；`xhs_web.py feed --bodies` 实测抓到真实笔记正文 |
| 小红书 评论 | ❌ 需登录 |
| 京东 评论 API（含预热 cookie） | ❌ 返回「系统繁忙」(反爬) |
| 淘宝 / 拼多多 | 封锁更严 |
| DuckDuckGo HTML / lite | ⚠️ 首次可用，几次后 202 反爬 → best-effort，不稳 |
| Bing HTML | ⚠️ 200 但结果走 JS，匿名抓不到结构化结果 |
| 搜索/数据 API（Brave/Tavily/Bing API） | ✅ 稳定（需 key，有免费层）——**采纳为真实来源主力** |

**一句话**：匿名免费抓取不稳；要「较稳地按商品拿到评论好坏」，靠**搜索 API（稳）+ 用户提供 + AI 先验**，自由抓取只作 best-effort 增强。

## 2. 要拍板的问题

§5.4 的口碑佐证，**主要数据从哪来**？这决定工程量、稳定性、合规风险和对评委的说法。

## 3. 候选方案

### 方案 A：实时爬虫为主（PRD v0.2 初始设想）
- 做法：用户给商品 → 实时去小红书/电商抓评论 → AI 提炼。
- 优点：最"真实"，故事性强。
- 缺点：实测已被封；要长期对抗反爬/签名/登录，三人三周扛不住；合规与"数据偏颇"风险高。

### 方案 B：分层取数（用户提供 + AI 先验为主，真实抓取为增强）★ 我的建议
- 分层、逐层降级、永不阻塞主流程：
  - **T0 用户提供**（地基）：截图(多模态/OCR)、链接公开页、用户粘贴评论。
  - **T1 AI 先验**（主力）：Claude 基于品类生成常见坑/吃灰率/适用人群/替代品。保底，永远产出。
  - **T2 真实口碑**（增强）：`xhs_web.py` 抓到笔记正文就叠加并标注来源；抓不到就降级。
  - **T3 第三方/官方授权 API**：比赛后。
  - **T4 本地登录态**：用户本地自带 Cookie 解锁搜索，绝不服务端存储，仅 Demo。
- 优点：永远有结果；稳；合规可控；对评委可说"多源+标注来源+优雅降级"。
- 缺点：T2 真实内容是机会性的，不保证每个商品都有。

### 方案 C：纯 AI 先验
- 只用 Claude 品类知识，不抓任何真实数据。
- 优点：最简单、零反爬零合规。
- 缺点：缺"真实佐证"卖点，容易被说"只是模型在讲道理"。

### 方案 D：用户提供为主
- 主要靠用户粘贴/截图评论，系统只做总结。
- 优点：最稳、最合规。
- 缺点：增加用户操作负担；没有"自动感"。

## 4. 关键判断（供参考，不是结论）

- 冷静购的目标不是"穷举评论"，而是"给一个能让人停一下的理由"——T1+T0 往往就够。真实抓取是锦上添花，不必在关键路径。
- 用户是在"看到商品、动了念头"的当下主动把商品带进来的 → 用户提供是天然主入口，不是兜底。

**我倾向方案 B**，但这是你们的决定。
